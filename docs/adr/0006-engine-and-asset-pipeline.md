# ADR 0006: вибір движка й конвеєра асетів (веб + AI/Blender, не Unity)

## Статус

Прийнято, 2026-07-11. Розглянуто й відхилено перехід на Unity після емпіричного тесту. Підстава — [технічний розтин конкурента](../research/dungeon-crusher-teardown.md) і прогін власного hero-конвеєра.

## Контекст

Референс Dungeon Crusher виглядає «дорого», і виникла гіпотеза, що для якісних об'ємних героїв треба перейти на Unity. Розтин показав: у нього це **Unity 6 → WebGL, ~60 MB перший вантаж, 512 MB heap**, а «об'єм» дають справжні 3D-моделі під ортокамерою + URP outline/bloom. Для Telegram Mini App вага Unity-порту вбиває миттєвий старт. Окремо стояло питання: чи може AI-агент автономно будувати в різних движках.

## Рішення

1. **Рантайм лишається веб-стеком проєкту:** `React + PixiJS` (shell/бій) та `Three.js` (skinned-3D вороги/герої, як у [ADR 0004](0004-layered-enemy-animation.md)). Unity не використовуємо для гри.
2. **Движок не створює персонажів — він їх показує.** Герої/вороги народжуються в **AI-генерації + Blender**, а не в движку. Це двигун-агностично й не є аргументом за Unity.
3. **Канонічний hero/asset-конвеєр:** Higgsfield `nano_banana_2` (2D) → `rembg` (вирізка) → Higgsfield `image_to_3d --should_texture [--enable_rigging]` (GLB) → Blender (перевірка/дотекстуринг/кліпи) → Three.js (`GLTFLoader`+`MeshoptDecoder`). Кроки, версії та пастки — у [розтині, §3](../research/dungeon-crusher-teardown.md). Кожен генеративний вихід документується provenance (як `art-source/rift/ironroot-3d/provenance.json`).
4. **Metallic-фікс для Three:** GLB із металевими матеріалами вимагає environment-мапи (`PMREMGenerator` + `RoomEnvironment`), інакше рендериться чорним. Застосовувати в 3D-сценах.
5. **Причина вибору веб-стеку для агентної розробки:** лише у вебі агент замикає повний автономний цикл «зібрав → побачив → виправив» (браузер: DOM/console/скрін/HMR). Емпіричний Unity-тест (`C:/AI Project/unity-idle-test`, поза репо): агент будує сцену/код/плеєр через CLI+C#, але **наосліп** (немає Game View), кожна ітерація — холодний старт движка, asset-importer чинить опір. Unity перетворює агента на C#-асистента, а не автономного виконавця.

## Наслідки

- Не витрачаємо зусиль на порт у Unity; наявні `RiftPixiScene`/`RiftThreeEnemyScene` — правильна основа.
- Якість героїв — це інвестиція в AI-арт-конвеєр і Blender-поліш, не в зміну движка. Наступний крок якості — дотекстуринг 3D та єдиний стиль ростера.
- Техніки конкурента (2.5D-скролер локації, фонтан монет за сумою золота, outline/bloom) переносимо на власний стек — не асети, а прийоми.
- `autosprite` (2D frame-анімація Higgsfield) headless нежиттєздатний; якщо знадобиться саме він — керувати веб-інтерфейсом через браузер, не CLI.
- Перед масштабуванням на десятки героїв потрібні спільний Blender export-profile, LOD-policy та перевірка ліцензії кожного генеративного постачальника (продовження [ADR 0004](0004-layered-enemy-animation.md)).
